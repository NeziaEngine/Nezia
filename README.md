# RESIA

![img.png](logi_resia.png)

データ指向設計によるサウンドミドルウェアライブラリ（Rust）。

## 特徴

- **スパースセット方式のデータ指向設計** — サウンドオブジェクトをコンポーネント単位で密配列に格納し、キャッシュフレンドリーな一括処理を実現する。
- **二層ID設計** — オーサリングツール向けの不変な論理IDと、ランタイム向けの高速な物理IDを分離する。
- **世代管理** — 物理IDに世代カウンタを持たせ、解放済みスロットへの誤アクセスを検出する。

## 設計思想

### なぜデータ指向か

サウンドミドルウェアは毎フレーム数百〜数千のボイス・バス・エフェクトを処理する。
オブジェクト指向的に各サウンドオブジェクトを個別のインスタンスとして扱うと、メモリアクセスが散在しキャッシュミスが増大する。

resia はスパースセット方式を採用し、同種のデータを連続メモリに配置することで、CPUキャッシュを最大限に活用する。

### 二層ID設計

サウンドオブジェクトの識別には 2 種類のIDを用いる。

#### 論理ID（Hash ID） — 「マイナンバー」

```
"MasterBus"  →  hash()  →  0x3F8B2A1C
```

- オーサリングツールやJSON等のアセットデータで使われる**不変の識別子**。
- 同じ文字列からは常に同じハッシュ値（`u32`）が生成される。
- ゲームを何度起動しても、アセットが存在する限り値は変わらない。
- 値域が広い（最大約42億）ため、配列のインデックスとしては使えない。

#### 物理ID（Entity ID） — 「座席番号」

```
Entity ID = (index: 5, generation: 2)
```

- スパースセットが内部で発行する**一時的な識別子**。
- `index` は 0 始まりの連番で、密配列への O(1) アクセスを可能にする。
- `generation` はスロット再利用時にインクリメントされる。古いハンドルが再利用後のスロットにアクセスしようとした場合、世代の不一致で無効と判定できる。
- ゲーム再起動やオブジェクト再生成で変わり得る。

#### 両者の関係

```
               ルックアップ (初期化時)
論理ID (Hash)  ────────────────────→  物理ID (Entity)
0x3F8B2A1C                            (index: 5, gen: 2)
                                           │
                                           ▼  O(1) アクセス (毎フレーム)
                                      dense[5] = データ
```

ランタイムでは物理IDのみで処理を回し、論理IDからの変換は初期化やイベント発火など低頻度の場面に限定する。

### スパースセットの構造

```
sparse (疎配列):  sparse[entity.index] → dense_index
dense  (密配列):  実データが隙間なく詰まっている
entities:         dense[i] に対応する Entity ID の逆引き
```

- **挿入**: sparse にマッピングを書き、dense 末尾にデータを追加。O(1)。
- **削除**: dense 末尾の要素と入れ替えて pop。O(1)。
- **検索**: sparse[index] で dense_index を取得し、dense[dense_index] にアクセス。O(1)。
- **走査**: dense 配列を先頭から舐めるだけ。キャッシュライン上で連続アクセス。

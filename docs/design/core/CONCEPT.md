# core クレート — コンセプト

## 位置づけ

`core` は NEZIA ENGINE のサウンドエンジン本体を提供するライブラリクレートである。
ゲームエンジン・オーサリングツール・上位ラッパー（`ffi`, `daemon`）から利用される、
プラットフォーム非依存のミドルウェアとして設計する。

## 設計思想

### データ指向設計（Data-Oriented Design）

オブジェクト指向的な「サウンドオブジェクトに振る舞いを持たせる」設計ではなく、
**データのレイアウトを最優先**して設計する。バス・Source・エフェクト等の状態は
コンポーネント種別ごとの密配列（dense array, SoA レイアウト）に格納し、
オーディオコールバック内で一括ループ処理することで CPU キャッシュ効率と
SIMD 適用性を最大化する。

### スパースセット方式の ECS

すべてのサウンドオブジェクトはスパースセットで管理する。
ハンドル（`EntityId`）からは O(1) で密配列インデックスに解決でき、
`generation` カウンタによってスロット再利用後の古いハンドルを安全に弾く。

詳細な ECS 設計は [ecs.md](ecs.md) を参照。

### 二層 ID 設計

| 層 | 用途 | 値域 | 永続性 |
|---|---|---|---|
| 論理 ID（Hash ID, `u32`） | アセット参照・永続化 | ハッシュ空間（~4G） | 不変 |
| 物理 ID（Entity ID） | ランタイムでの高速アクセス | `0..MAX` の連番 | 一時的 |

オーサリングデータ（JSON 等）は論理 ID を持ち、ランタイム初期化時に
`HashMap<HashId, EntityId>` を構築する。フレーム処理では物理 ID のみを使い、
ハッシュ検索は初期化・イベント発火時に限定する。

### スレッド分離とリアルタイム制約

サウンドスレッド（オーディオコールバック）はリアルタイム制約下にあり、
**ロック・確保・syscall を一切行わない**。メインスレッドからの操作は
ロックフリーなコマンドキュー経由で伝達し、結果通知はイベントリングバッファで
返す。詳細は [threading.md](threading.md) を参照。

## モジュール構成

| モジュール | 責務 | 設計ドキュメント |
|---|---|---|
| `bus` | バスの木構造ルーティング・ミキシング | [bus.md](bus.md) |
| `source` | サウンドソース（発音体）の管理 | [source.md](source.md) |
| `spatial` | 3D 空間音響（距離減衰・パンニング・ドップラー） | [spatial.md](spatial.md) |
| `event` | コールバック用イベント通知 | [callbacks.md](callbacks.md) |
| `core` | エンジン中核（`SoundEngine`, スパースセット, ルーティング） | — |

## 公開 API の方針

`core` クレートは **ミドルウェアとしての最小公開面** を保つ。
`SoundEngine`, `BufferId`, `EntityId`, `AttenuationModel` 等のごく少数の型のみを
`pub` 公開し、内部モジュール（`bus`, `source` 等）の実装詳細は隠蔽する。
上位レイヤー（`ffi`, `daemon`）はこの最小 API を経由してエンジンを操作する。
